## What's done

The outer layers are converted to ThreadId:

- **`thread_metadata_store.rs`**: `ThreadId` type, `reserve_thread_id()`,
  `thread_id_for_session()`, `handle_thread_event` checks session reservations
  before generating new IDs.
- **`agent_panel.rs` public API**: `load_agent_thread` takes `ThreadId`,
  `cancel_thread` takes `ThreadId`, `MessageSentOrQueued` event carries `ThreadId`,
  `open_thread` accepts `SessionId` (external boundary) but looks up ThreadId internally.
- **`sidebar.rs`**: All tracking structures use `ThreadId` via the metadata store.
  Uses `metadata.thread_id` for sorting, notifications, active entry tracking.
  Calls `panel.load_agent_thread(metadata.thread_id, ...)`.
- **`sidebar_tests.rs`**: Updated for ThreadId, all 88 tests pass.
- **`mention_crease.rs`**: Looks up ThreadId from metadata store at the SessionId boundary.

## What remains

The **inner agent_ui layer** still uses `SessionId` for internal tracking where
it should use `ThreadId`. SessionId should ONLY remain where it directly touches
the ACP protocol (reading `thread.session_id()`, calling `connection.load_session()`,
`connection.resume_session()`, `connection.model_selector()`, etc.).

### ThreadId creation discipline

`ThreadId::new()` should ONLY be called when genuinely creating a new thread:

- `create_agent_thread` fallback when no metadata exists for `resume_session_id` — correct
- `load_subagent_session` for new subagent threads — correct
- `thread_import.rs` for imported threads — correct
- `handle_thread_event` last-resort fallback — acceptable but should rarely fire
  if reservations are working

It should NEVER be called as a "I don't know the ID so I'll make one up" fallback.
If you're tempted to call `ThreadId::new()` somewhere that isn't creating a brand new
thread, that's a bug — you need to look up or pass through the existing ID.
